## What this epic is (and what it is NOT)

This is the **second half of Phase 3**: the guest-assembled **cross-business cart** (Q32R — a guest puts reservables from **one OR many independent tenants** into one cart "at any time" and checks out atomically). The **first half** (VRB-40x/41x) already landed the machinery this epic *consumes*: the `Order`/`Reservation` split, the `ordering` schema, the `app.user_id` GUC, `ReservableKind`/`ReservableId` polymorphism, and count-based room availability. This epic does **not** re-invent any of that — it wires the *checkout* concerns on top: multi-line assembly, atomic multi-tenant Place, split payment, cross-N manual capture, per-item cancel, FX, mixed display, and the cart/checkout UI.

**Locked design principle (2026-07-13):** *standardize the framework/machinery (policy engine, tax model, currency-display, split-payment), localize the values per-property.* That is precisely what lets a cross-business cart span autonomous tenants without forcing them onto identical rules (Q39R).

## Writing to the CORRECTED design — how the three blockers are storied

The independent architect review (`PHASE-3-4-DESIGN-REVIEW.md`) found three **code-verified blockers** in the original §3/§4 prose. This epic is written to the **corrected** model (§0.5), and each blocker is a first-class story so the fix is testable, not buried:

- **C3 (RLS write under EF batching)** → **VRB-422**. The original claim "the interceptor re-stamps `app.tenant_id` per statement" is FALSE: `TenantGucCommandInterceptor` fires **per `DbCommand`** (`TenantGucCommandInterceptor.cs:70-106`, `set_config('app.tenant_id', …, true)` at :87-89), and EF **batches** N INSERTs into ONE command, so a mixed-tenant batch stamps a single tenant and the other tenant's rows fail their `WITH CHECK`. Corrected: the atomic N-line Place **flushes per tenant** (N `SaveChanges`, one `BackgroundTenantScope` each) **within one serializable transaction**.
- **C2 (PaymentIntent is a SECOND cross-tenant object)** → **VRB-423**. The headline "the Order is the ONLY cross-tenant object" is FALSE: an `OrderId`-keyed PI spans M suppliers, but `payment.payment_intents` is tenant-scoped today (RLS + the `pi.TenantId != cmd.TenantId` guard at `RefundForBookingCommand.cs:65`). Corrected: the PI root moves to **platform/orchestration scope** (read/written via `IRlsBypassDbContextFactory`; only the platform is Merchant-of-Record); `payment.transfers` + refunds carry the per-tenant axis.
- **C1 (multi-supplier manual capture)** → **VRB-425**. "Capture-on-first-confirm" is REJECTED — a Stripe PI captures **once**, so first-confirm would charge the guest for legs suppliers B…N haven't approved (violates Q6 manual capture). Corrected: **resolve all legs by the 48h SLA** (confirm/reject/expire) → **ONE partial capture** = Σ(confirmed legs) via `AmountToCapture` → **transfer per confirmed supplier**.

C4/C5 (fee-reversal + MoR) are **launch** fixes owned by the sibling launch epic (VRB-104/VRB-105) and are hard **blocked-by** dependencies here — a cross-tenant per-item refund is only financially correct once the real `ApplicationFeeRefundService` reversal exists. C6/C7/C8/C9/C10/C11 are folded into the stories below as noted.

### VRB-420 — Cross-business Order draft assembly: add-item re-probe + re-quote across 1..M tenants
- **Epic:** Phase 3 — Multi-unit & Cross-business Cart · **Priority:** Could (post-launch) · **Estimate:** L
- **Narrative:** As a **guest**, I want to add reservables from one or many independent businesses to a single cart at any time, so that I can plan and buy a multi-property (and later multi-supplier) trip in one checkout instead of paying each business separately.
- **Acceptance criteria:**
  - **Given** an authenticated guest, **when** they create a cart, **then** a `Draft` `ordering.orders` row is persisted with `GuestUserId`, `Reference`, `DisplayCurrency`, `Status=Draft`, `OriginatingTenantId=null` (guest cart, not agency), and an empty `_reservations` set — **no Money total on the root** (totals are computed projections, §1.2/§5).
  - **Given** a `Draft` order, **when** the guest adds an item `(ReservableKind, ReservableId)` — from the **same tenant OR a different tenant than existing lines** (Q32R "at any time") — **then** the matching `IReservableResolver` (from VRB-41x) is invoked and a `Reservation` line is appended carrying `TenantId` (the supplier), `ReservableTitleSnapshot`, `Stay?`, `GuestCount`, its **own `SettlementCurrency`**, its **resolved `RatePlan`/`CancellationPolicy` snapshot** (C7 — plan = price × policy × prepayment, snapshotted immutable at add), a **price quote snapshot**, and a **`TaxSnapshot`** (VRB-429) — all **per line**.
  - **Given** an add-item request, **when** the resolver runs, **then** it **re-probes availability** (count-based for rooms per VRB-41x) and **re-quotes** price+plan+tax at that moment; a stale/unavailable reservable returns `ordering.item_unavailable` and the line is **not** added (the cart never holds a phantom line).
  - **Given** a line already in the cart, **when** the guest re-opens the cart after time has passed, **then** each line's availability + quote is **re-validated on read** and any drifted/expired line is flagged stale in the DTO (not silently priced at the old number) — the authoritative re-probe happens again at Place (VRB-422).
  - **Given** a cart with lines from M distinct tenants, **when** `DistinctTenantCount()` is evaluated, **then** it returns M (drives the VRB-424 payment selector).
  - **Invariant:** a `Reservation` line is **tenant-scoped** (`booking.reservations`, RLS by `TenantId`); the `Order` root is **guest-scoped** (`ordering.orders`, no `TenantId`). Adding a cross-tenant line never writes a "tenant A OR tenant B" row anywhere.
  - **API:** `POST /orders`, `POST /orders/{id}/items`, `DELETE /orders/{id}/items/{reservationId}`, `GET /orders/{id}`, `GET /me/orders`. Existing `bookings/*` endpoints alias during transition (one release).
- **TDD plan:**
  - *Unit:* `OrderDraftTests.Add_item_appends_reservation_line_with_own_currency_and_policy`; `..._distinct_tenant_count_counts_M_tenants`; `..._no_money_total_on_root`; `..._add_item_snapshots_rate_plan_immutably`; `..._unavailable_reservable_rejects_line`.
  - *Integration (Testcontainers Postgres):* `OrderAssemblyRlsTests.Draft_order_is_guest_scoped_reservation_lines_are_tenant_scoped`; `CrossTenantCartTests.Add_lines_from_two_tenants_reprobe_and_requote_each` (opens a `BackgroundTenantScope` per tenant to resolve, `MyBookingsHandler.cs:41-55` idiom); `..._reopen_cart_flags_stale_line`.
  - *E2E (Playwright):* `cart-assemble-cross-business.e2e` (guest adds a room from tenant A + a whole-house from tenant B → cart shows 2 lines, 2 currencies, 2 policies).
- **Technical notes:** New `AddOrderItemHandler` / `CreateOrderHandler` / `RemoveOrderItemHandler` in `Modules.Ordering` (or `Booking`-adjacent per VRB-40x placement). Add-item resolves via the VRB-41x `IReservableResolver` registry returning `(title, settlementCurrency, availabilityProbe, priceQuote)`; the re-quote also resolves the **RatePlan** (C7) + **CancellationPolicy** VO snapshot (§6) + **TaxSnapshot** (VRB-429). Reads across tenants reuse the per-tenant-scope-and-merge primitive (`MyBookingsHandler.cs:41-55`). `Money.Add` throwing on cross-currency (`Pricing/Domain/Money.cs:29`) is **relied on** — the cart holds a *collection* of per-line `Money`, never a summed root total. Events: `OrderDrafted`, `OrderItemAdded`, `OrderItemRemoved`. No new schema beyond what VRB-40x ships (`ordering.orders`, `booking.reservations`); this story is behavior on those tables.
- **UI/UX spec:** Backend/aggregate story; the cart surface is VRB-430. DTO MUST expose per-line `tenantDisplayName`, `settlementCurrency`, `policySummary`, `isStale`, and the order-level `displayCurrency` + `distinctTenantCount` so VRB-430/428 render deterministically without a second round-trip.
- **Configuration:** `Cart:Enabled` (default **false** all envs). `Cart:MaxLinesPerOrder` (default 12). `Cart:MaxDistinctTenants` (default 6). `Cart:StaleLineToleranceMinutes` (default 30 — re-quote drift window before a line is flagged stale). No secrets.
- **Rollout:** Feature flag `Cart:Enabled=false` at merge everywhere. Migration order: after all VRB-40x/41x migrations (`ordering` schema + `ReservableKind` must exist). Backward-compat: single-line same-tenant orders are behaviorally identical to today's booking; `bookings/*` alias one release. Rollback: flag off; drafts are inert (nothing placed).
- **Observability:** Log `order_id`, `guest_user_id`, `line_count`, `distinct_tenant_count` on draft/add/remove. Metric `cart.item.added{cross_tenant=true|false}`, `cart.item.rejected_unavailable`. Track add→place funnel drop-off.
- **Definition of Done:** unit+integration green → architect review of resolver re-quote path → staging: guest assembles a 2-tenant cart, each line re-probed+re-quoted → prod (flag on for pilot) → monitored 1 week.
- **Dependencies:** **blocked-by** VRB-40x (Order/Reservation split, `ordering` schema, `app.user_id` GUC), VRB-41x (`IReservableResolver`, room count availability), and VRB-102 (CancellationPolicy VO for the per-line snapshot). **blocks** VRB-421, VRB-422, VRB-427, VRB-428, VRB-429, VRB-430.
- **Parallelisation:** Lane **A (ordering-domain)**. Owns `Modules.Ordering/Application/Commands/*OrderItem*`, the Order aggregate assembly methods, and the add-item resolver-dispatch glue. Does **not** touch `StripeGateway.cs`/`PlaceBookingHandler.cs`/`TenantGucCommandInterceptor.cs` (those are the sequenced VRB-422/424/425 lane).

### VRB-421 — Anonymous-cart posture: client-side cart until auth; checkout forces sign-in
- **Epic:** Phase 3 — Multi-unit & Cross-business Cart · **Priority:** Could (post-launch) · **Estimate:** M
- **Narrative:** As a **not-yet-signed-in visitor**, I want to browse and collect reservables before I have an account, so that I only sign in when I actually check out — without the platform needing to store a server-side cart it has no owner axis for.
- **Acceptance criteria (correction C11):**
  - **Given** the guest-scoped RLS model, **when** a visitor is anonymous (`app.user_id` empty), **then** **no server-side `ordering.orders` row is created** — the anonymous `app.user_id` GUC **fails-safe-denies** any `ordering.orders` read/write (re-verifies Q5). There is **no phantom guest** and no orphan cart.
  - **Given** an anonymous visitor, **when** they add items, **then** the cart is held **client-side only** (browser storage) as an intent list of `(tenantId, reservableKind, reservableId, stay, guestCount)` tuples — **no prices are trusted from the client**; prices/policies/tax are always re-resolved server-side at first authenticated `POST /orders/items`.
  - **Given** a client-side cart, **when** the visitor initiates checkout, **then** they are **forced through sign-in/sign-up** (guest MSAL flow — email+password OR social per the owner-locked guest IdP surface); after auth the client-side tuples are **replayed** into a server-side `Draft` order (each tuple re-probed + re-quoted via VRB-420) and the client cart is cleared.
  - **Given** replay, **when** one tuple is now unavailable or re-prices, **then** the guest is shown the delta (removed/changed lines) **before** Place — no silent substitution.
  - **Invariant:** the server never persists an unauthenticated cart; the only durable cart is a signed-in guest's `Draft` order. This is a direct consequence of guest-scoped RLS (§4), not a UX preference.
- **TDD plan:**
  - *Unit (web, Vitest):* `anonymousCart.store.test.ts` (add/remove/persist tuples, no price fields trusted); `checkoutForcesAuth.test.ts` (checkout when anonymous → redirect to sign-in with return-to-cart).
  - *Unit (api):* `OrderingRlsDenyTests.Anonymous_user_id_denies_ordering_orders` (empty `app.user_id` → RLS deny, re-verifies Q5 fail-safe).
  - *Integration:* `CartReplayTests.Signin_replays_client_tuples_into_draft_order_requoting_each`; `..._unavailable_tuple_surfaces_delta_not_silent_drop`.
  - *E2E (Playwright):* `anon-cart-to-auth-handoff.e2e` (anonymous add 2 items → click checkout → sign in → server draft has 2 re-quoted lines → one drifted line flagged).
- **Technical notes:** Client cart lives in a typed web store (`web/src/features/cart/anonymousCart.ts`) persisted to `localStorage` with a schema version + TTL. Server side reuses VRB-420's `POST /orders/items` for replay; a thin `POST /orders/from-intent` batch endpoint replays N tuples in one call, each re-quoted. RLS deny confirmed against the `app.user_id` GUC added by VRB-40x in `TenantGucCommandInterceptor` (additive `set_config('app.user_id', …)` alongside `app.tenant_id` at `TenantGucCommandInterceptor.cs:87-89`). No card data, no PII in `localStorage` (only reservable references).
- **UI/UX spec:** Anonymous cart badge with item count in the header; cart drawer works signed-out. "Sign in to check out" CTA at the checkout boundary with a clear return-to-cart. Post-auth "we re-checked your cart" reconciliation panel showing kept/removed/re-priced lines. a11y: focus returns to the reconciliation panel after the auth round-trip; announce line-count changes via `aria-live=polite`. Respect `prefers-reduced-motion` on the drawer.
- **Configuration:** `Cart:AnonymousEnabled` (default **false**). `Cart:AnonymousTtlHours` (default 72 — client cart expiry). Under `Cart:Enabled`.
- **Rollout:** Flag off at merge. No schema. Backward-compat: with the flag off, checkout simply requires an existing sign-in (today's behavior). Rollback: flag off; `localStorage` carts are inert and TTL-expire.
- **Observability:** Metric `cart.anonymous.created`, `cart.anonymous.replayed`, `cart.replay.line_dropped`, `cart.checkout.auth_redirect`. Log replay deltas (kept/removed/re-priced counts) — the conversion-risk surface.
- **Definition of Done:** unit+integration+e2e green → review (confirm no server-side anonymous persistence path exists) → staging: anon→auth handoff walk → prod pilot → monitored (replay drop-off).
- **Dependencies:** **blocked-by** VRB-420, VRB-40x (`app.user_id` GUC + fail-safe-deny). **blocks** VRB-430 (checkout UI consumes the handoff).
- **Parallelisation:** Lane **F (web-cart)**. Owns `web/src/features/cart/*` + the `from-intent` replay endpoint. Independent of the payments lane.

### VRB-422 — Atomic PlaceOrder: per-tenant flush (N `SaveChanges`) within ONE serializable transaction
- **Epic:** Phase 3 — Multi-unit & Cross-business Cart · **Priority:** Could (post-launch) · **Estimate:** L
- **Narrative:** As a **guest**, I want my whole cart to be reserved all-or-nothing, so that I never end up paying for half a trip when one property sells out at the moment I check out.
- **Acceptance criteria (correction C3 + Q34):**
  - **Given** a `Draft` order spanning M tenants with N reservation lines, **when** the guest places it, **then** all N lines are inserted inside **ONE serializable transaction**, but the interceptor's per-tenant GUC is honored by **flushing per tenant** — i.e. **N (well, M) `SaveChanges` calls, each inside its own `BackgroundTenantScope(tenantId)` frame** — so each INSERT batch is stamped with the correct `app.tenant_id` and passes **its own** `WITH CHECK` policy.
  - **Given** EF batches multiple INSERTs into one `DbCommand`, **when** a mixed-tenant flush is attempted in a single `SaveChanges`, **then** the design MUST NOT do that (it would stamp one tenant and fail the other tenant's rows) — the test suite **proves** the per-tenant-flush requirement by asserting a naïve single-flush mixed-tenant insert is rejected, and the corrected per-tenant-flush path succeeds.
  - **Given** the atomic Place, **when** each line's availability is re-checked, **then** the existing race-safe primary check generalizes per line: `SELECT … FOR UPDATE` over `booking.reservations` for stays (`PlaceBookingHandler.cs:152-180`) and the **room-inventory counter row `FOR UPDATE`** for room lines (correction C8 — lock `booking.room_inventory`, never a phantom-row `COUNT`), all inside the one serializable txn.
  - **Given** any single line is unavailable, **when** Place runs, **then** the **whole transaction rolls back**, **nothing is reserved**, and the response is **409 identifying the failed line** (`reservationIndex` + `reservableId` + `tenantId`) — the `40001 → ConflictException` map (`PlaceBookingHandler.cs:217-225`) generalizes to "another guest just booked line K."
  - **Given** a successful Place, **when** the txn commits, **then** every line is `Tentative` with `PlacedTentativeUntil = now + 48h` (correction C10 — 48h everywhere, depends on the G2 fix that removes the hardcoded 24h at `Booking.cs:119`), the order is `Placed`, and `OrderPlaced` + one `ReservationPlaced` per line are emitted (`BookingPlaced` kept one release for back-compat).
  - **Invariant (test-named):** *"a mixed-tenant Order Place flushes per tenant within one serializable txn; each reservation passes its own RLS `WITH CHECK`; a partial-availability Place reserves zero lines."*
- **TDD plan:**
  - *Unit:* `PlaceOrderPlanTests.Groups_lines_by_tenant_for_per_tenant_flush`; `..._sets_48h_tentative_window`.
  - *Integration (Testcontainers Postgres — the load-bearing suite):* `AtomicPlaceRlsTests.Mixed_tenant_batch_single_flush_is_rejected_by_with_check` (proves the C3 defect exists under EF batching); `AtomicPlaceRlsTests.Per_tenant_flush_in_one_serializable_txn_succeeds`; `AtomicPlaceTests.One_unavailable_line_rolls_back_all_and_returns_409_identifying_line`; `AtomicPlaceTests.Room_line_locks_room_inventory_counter_for_update` (C8 overbooking-at-boundary); `AtomicPlaceTests.Concurrent_place_on_same_dates_maps_40001_to_409`.
  - *E2E (Playwright):* `place-order-atomic.e2e` (2-tenant cart → one property blocked mid-checkout → 409 names the line → nothing reserved → retry after removing it succeeds).
- **Technical notes:** Reshape `PlaceBookingHandler` → `PlaceOrderHandler` (`src/Modules/VrBook.Modules.Booking/Application/Commands/PlaceBookingHandler.cs`). Keep the outer `IDbContextTransaction` at **serializable** isolation; **group lines by `TenantId`** and, per group, enter `BackgroundTenantScope(tenantId)` → run the `FOR UPDATE` availability probes (stay overlap SQL at :152-163; block SQL at :185-192; **new** `booking.room_inventory` counter lock for rooms) → `AddAsync` the group's reservations → `SaveChangesAsync` (one flush per tenant so the interceptor stamps that tenant's GUC). All groups share the one transaction; commit once at the end; `40001 → ConflictException` (:217-225). The PI is created **outside** this txn (unchanged pattern, :227+) and is now platform-scoped per VRB-423. `TenantGucCommandInterceptor` needs **no change** for this (its per-`DbCommand` stamp is exactly what per-tenant-flush relies on) — the fix is in the *handler's flush boundaries*, not the interceptor.
- **UI/UX spec:** The 409-identifying-line response drives VRB-430's inline "this item just sold out" line-level error with a "remove & retry" affordance; no full-cart wipe. a11y: the failed line gets `aria-invalid` + an `aria-describedby` error; focus moves to it.
- **Configuration:** `Cart:PlaceMaxRetries` (default 1 — one automatic retry on `40001` before surfacing 409). `Booking:TentativeSlaHours` (default **48**, honoring the G2 fix — this key must actually be read, not the hardcoded 24h). Under `Cart:Enabled`.
- **Rollout:** Flag off at merge. Migration order: needs `booking.room_inventory` (from VRB-41x) present. Backward-compat: a single-line single-tenant Place is the exact today-path (one tenant group, one flush). Rollback: flag off; single bookings unaffected.
- **Observability:** Metric `order.place.atomic{outcome=committed|rolled_back}`, `order.place.conflict_409{reason}`, `order.place.serialization_retry`. Log `order_id`, `tenant_group_count`, `line_count`, `failed_line_index` on rollback. **Alert:** overbooking-at-place (any committed order where a post-hoc audit finds two active reservations over the same inventory-unit/date — should be zero given `FOR UPDATE`; fire if > 0).
- **Definition of Done:** unit+integration green (the C3 defect + fix pair MUST both be asserted) → architect review of flush boundaries → staging: 2-tenant atomic place + forced-conflict rollback → prod pilot → monitored (zero overbooking).
- **Dependencies:** **blocked-by** VRB-420, VRB-41x (`booking.room_inventory` for C8), and the G2 48h fix (C10). **blocks** VRB-423, VRB-424.
- **Parallelisation:** Lane **P (payments/checkout — SEQUENCED)**. Owns `PlaceBookingHandler.cs` → `PlaceOrderHandler`. **Must land before** VRB-423/424/425 which build on the placed-order + PI shape. Do NOT parallelize with those.

### VRB-423 — PaymentIntent re-scoped to platform/orchestration; re-key `BookingId → OrderId` (the SECOND cross-tenant object)
- **Epic:** Phase 3 — Multi-unit & Cross-business Cart · **Priority:** Could (post-launch) · **Estimate:** L
- **Narrative:** As the **platform**, I want the PaymentIntent that spans a multi-supplier order to live at platform/orchestration scope, so that one guest charge can legitimately reference M suppliers without any tenant's RLS blocking it and without pretending the Order is the only cross-tenant object.
- **Acceptance criteria (correction C2):**
  - **Given** the review's finding that `payment.payment_intents` is **tenant-scoped today** (RLS + the `pi.TenantId != cmd.TenantId` defense-in-depth guard at `RefundForBookingCommand.cs:65`), **when** an `OrderId`-keyed PI spans M suppliers, **then** the PI root is **re-scoped to platform/orchestration** — read/written only via the audited, allowlisted `IRlsBypassDbContextFactory` (`RlsBypassScope`), because the **platform is the sole Merchant-of-Record** for the guest charge (C5/Q25).
  - **Given** the PI re-scoping, **when** the schema is migrated, **then** `payment.payment_intents` is re-keyed **`BookingId → OrderId`** (one PI per Order, not per Booking); the per-tenant axis moves to `payment.transfers` (VRB-424) + `payment.refunds`.
