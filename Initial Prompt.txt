## PROJECT BACKGROUND

The client owns and operates several short-term rental properties in the **United States**, currently listed exclusively on AirBnB. AirBnB's host commission (typically ~3% host fee plus the guest service fee that suppresses headline price competitiveness) is eroding margin. The client will **continue listing on AirBnB** but wants a **direct-booking website** so repeat guests and word-of-mouth referrals can book without paying AirBnB's markup.

Critical constraint: **the direct site must never double-book a property that is already booked on AirBnB** (or vice versa).

---

## BUSINESS GOALS (in priority order)

1. **Eliminate commission leakage** on direct/repeat bookings.
2. **Zero double-bookings** between AirBnB and the direct site.
3. **Launch Phase 1 (web) within 8 weeks.**
4. Build a foundation that supports **future channels** (native mobile apps, additional OTA channels like VRBO/Booking.com) without re-architecting.
5. Provide the owner with a **single operational dashboard** to manage properties, bookings, pricing, reviews, and messaging.

## PHASE 1 SCOPE — FUNCTIONAL REQUIREMENTS

All of the following must be in Phase 1, live within 8 weeks.

### 1. Property Catalog
- Property listings with: title, description, address (geocoded), property type, capacity (guests/bedrooms/bathrooms/beds), amenities (checklist), house rules, check-in/check-out times, photo gallery (multiple images, primary image, reordering), location map.
- Owner-side CRUD with image upload to Azure Blob Storage; image processing (resize, WebP conversion, thumbnails).

### 2. Search & Discovery
- Public search by destination, check-in/check-out dates, guest count.
- Filters: price range, amenities, property type, minimum rating.
- Map view with pins.
- Sort: relevance, price asc/desc, rating, newest.
- Availability-aware search (only show properties available for the selected dates).

### 3. Booking Flow (the heart of the system)
- **Hybrid availability strategy**:
  - **Primary**: Two-way iCal sync with AirBnB (poll AirBnB's exported `.ics` feed for each property on a configurable schedule; expose this platform's own `.ics` feed back to AirBnB for import). Default poll interval: 30 minutes; configurable per property.
  - **Safety belt**: Every direct booking is created in a `Tentative` state, NOT `Confirmed`. A reconciliation worker re-checks the AirBnB iCal feed before transitioning `Tentative → Confirmed`. The owner also receives a notification and can manually confirm/reject within a configurable SLA window (default 24h). If the SLA expires without manual action AND iCal reconciliation shows no conflict, auto-confirm.
- **Booking state machine** (must be explicitly designed): `Draft → Tentative → Confirmed → CheckedIn → CheckedOut → Completed`, with side-branches `Rejected`, `Cancelled`, `Refunded`, `Disputed`.
- Hold/lock dates during checkout (e.g., 15-minute reservation lock) to prevent race conditions on simultaneous bookings.
- Booking detail page with cancellation policy display.
- Cancellation flow with policy-driven refund calculation.

### 4. Payments (Stripe end-to-end)
- Stripe Payment Intents for full payment at booking, OR deposit + balance (configurable per property by admin).
- Stripe Customer object per registered guest; saved payment methods (PCI handled by Stripe).
- Refund handling (full, partial, policy-driven).
- Webhook handler for `payment_intent.succeeded`, `charge.refunded`, `charge.dispute.created`, etc.
- Idempotency keys on all Stripe calls.
- Owner payout configuration (Stripe Connect Standard or Express — propose with trade-offs).
- Tax handling: research and propose approach for US state/local lodging taxes (Stripe Tax vs. manual configuration per property/jurisdiction).

### 5. Guest Accounts
- Open registration: email/password + social (Google at minimum).
- Use **Azure AD B2C** for identity. Email verification required.
- Profile: contact info, saved payment methods (via Stripe), booking history, messages.
- Password reset, account deletion (GDPR/CCPA-friendly even though out of strict scope).

### 6. Owner Admin Dashboard
- Property management (CRUD + photo management).
- Calendar view per property showing AirBnB + direct bookings color-coded.
- Booking queue: pending tentative bookings requiring manual confirmation.
- Manual booking creation (for walk-ins / phone bookings).
- Pricing management:
  - Base nightly rate, weekend rate, cleaning fee, additional guest fee, security deposit, taxes.
  - **Dynamic/seasonal pricing rules** (date-range overrides, day-of-week multipliers, length-of-stay discounts).
- **Feature toggles per property and globally**: enable/disable reviews, enable/disable dynamic pricing, enable/disable in-app messaging.
- Reports: occupancy, revenue, average daily rate (ADR), booking source (direct vs. AirBnB).
- Review moderation queue.
- Guest list with booking history and loyalty tier.
- iCal feed URLs (in/out) per property with copy-to-clipboard.

### 7. Reviews & Ratings
- Post-stay review (1–5 stars + written) from guest, with owner response capability.
- Aggregate rating per property.
- Admin toggle to enable/disable reviews globally or per property.
- Moderation flow (auto-publish vs. admin-approval, configurable).

### 8. Dynamic / Seasonal Pricing
- Rule engine: priority-ordered rules (date range overrides, day-of-week multipliers, last-minute discounts, length-of-stay tiers, minimum stay enforcement).
- Preview tool: show calculated price for any sample date range.
- Admin toggle (per property, global default).

### 9. In-App Messaging
- Thread per booking between guest and owner.
- Real-time delivery via **Azure SignalR Service** (Serverless mode or default — propose).
- Read receipts, typing indicators (nice-to-have).
- Push (in-app + email fallback if recipient offline > N minutes).
- Admin can view all threads (moderation/support).
- Attachments (images) via Blob Storage.

### 10. Loyalty / Repeat Customer Logic
- Auto-tracked tier based on number of completed stays (e.g., Bronze 1–2, Silver 3–5, Gold 6+).
- Tier benefit: configurable % discount applied at checkout (admin sets % per tier).
- Loyalty enabled/disabled via admin toggle.
- Display tier in guest profile + at checkout.

### 11. Transactional Emails
Comprehensive catalog (all via SendGrid or Azure Communication Services — propose):
- Email verification
- Password reset
- Booking received (tentative)
- Booking confirmed
- Booking rejected
- Cancellation confirmation + refund notice
- Pre-arrival reminder (T-3 days, T-1 day)
- Check-in instructions (T-0)
- Post-stay review request (T+1 day after checkout)
- Owner: new tentative booking alert
- Owner: booking auto-confirmed (SLA elapsed, no conflict)
- Owner: cancellation alert
- Owner: dispute/refund alert
- Admin: iCal sync failure alert

Use a templating system (Mustache or Razor) with a shared layout, consistent brand styling, Outlook-safe MSO conditional comments.

### 12. iCal Sync Engine
- **Inbound**: Background worker (Azure Container App Job or hosted service) polls each property's AirBnB `.ics` URL on a schedule, parses with `Ical.Net`, reconciles into an `ExternalReservation` table, and flags conflicts.
- **Outbound**: HTTP endpoint that emits a valid `.ics` feed per property containing all `Confirmed` (and optionally `Tentative`) direct bookings, for AirBnB to import.
- Conflict detection: any overlap between an external reservation and a tentative/confirmed direct booking → flag for owner review.
- Health monitoring: track last-success time per property; alert if stale beyond threshold.

- **Frontend-agnostic backend**: no server-rendered HTML coupling; SPA + API split.
- **Performance**: search response P95 < 500ms; booking creation P95 < 1s; iCal poll worker concurrent across properties.

- **Security**: OWASP Top 10 compliance, HTTPS everywhere, secrets in Key Vault, no PII in logs, Stripe handles PCI scope.
- **Observability**: structured logging (Serilog → App Insights), distributed tracing, metrics on booking funnel, alerts on critical failures (payment, sync, auth).
- **Scalability**: stateless app tier, horizontal scale via Container Apps; background workers as separate Container App Jobs.
- **Testability**: unit test coverage ≥ 70% on Application + Domain layers; integration tests for critical flows (booking, payment, sync).

## AIRBNB SYNC STRATEGY (FINALIZED)

**Option B: iCal two-way sync PLUS tentative-booking safety belt.**

Design must explicitly cover:
1. Inbound iCal poll frequency, parsing, idempotency, dedup of events across polls.
2. Outbound iCal feed format, caching, ETag/Last-Modified support.
3. Lag-window handling: time between a booking being made on AirBnB and appearing in the exported feed (assume 1–2 hours worst case). The tentative-state grace period must cover this.
4. Conflict detection algorithm and owner notification pathway.
5. Failure modes: AirBnB feed unreachable, malformed, or stale — degraded behaviour and alerting.
6. Extensibility hooks so VRBO/Booking.com channels can be added later via the same `IExternalChannel` abstraction.
